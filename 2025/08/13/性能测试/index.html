基本概念

如何衡量一个接口的性能

在 4 核 8g 的服务器上，20 并发下，我们测试得到的 TPS 为 200，MRT（平均响应时间）为 150ms，RT90 值为 250ms，CPU 使用率为 65%

衡量指标

  • 并发数
  • TPS：系统吞吐量，每秒事物处理量（TransactionPerSencond），一个表达系统处理 能力的性能指标，每秒处理的消息数
  • MRT（平均响应时间）、RT90 值（90% 的响应时间都在这个值之下），RT99 值（99% 的响应时间都在这个值之下）这两个值因为每个系统的衡量指标不同，所有会有不同的形式，本质相同
  • 资源使用率：CPU、内存、网络、磁盘
  • 吞吐量：每秒钟处理成功的事物数／请求数
  • 衡量指标
    • TPS：Transaction Per Sencond
    • QPS：Query Per Second
    • IOPS：IO Per Second
    • BPS：Byte Per Second
  • 响应时间
    • 从客户端发起一个请求开始，到客户端接受到服务端的返回的最后一个字节结束，这个过程所耗费的时间
    • 响应时间 =（N1+N2+N3+N4）+（A1+A2+A3）
      当看到一个接口返回的特别慢的时候，我们如果要做到准确的去判断到底是哪个节点出量问题，就需要去打点，在每条线上，看是哪条线的响应时间最长，最后来对症下药
  • MRT（Mean Response Time）：平均响应时间
  • STD、DEV：标准差，响应时间波动，波动越小越好，趋于一条直线
  • 90%RT：90% 的响应时间小于该值，产品响应及时要求不同，如网络延迟及时性要在 99.99% 在某值范围内
  • MAX：最大值，长尾时间，最慢的请求的那个耗时
  • 资源使用率
    • 资源利用率指的是对不同系统资源的使用程度
    • 系统资源：CPU、内存，磁盘 IO、网络 IO
    • 应用资源：队列，锁，线程，GC，连接数等
  • 四个常见的性能测试的概念

如何开展性能测试

明确测试目标是什么，再根据测试目标去选择测试类型

1）性能测试目标和性能测试类型选择

  • 上线功能是否有性能问题
    • 评估系统性能：性能测试、负载测试
    • 检查系统性能瓶颈点：负载测试、压力测试
  • 优化后的接口，是否有性能提升
    • 回归测试、负载测试、压力测试
  • 验证稳定性、可靠性等：异常测试、稳定性测试
  • 容量评估：线上压测

2）明确哪些接口需要进行性能测试

  • 核心功能：譬如交易系统，订单支付等核心接口需要进行性能测试
  • 高频调用接口：例如登陆接口，或者从线上监控查看，每天哪些接口的调用量很大，从这里入手去测
  • 业务逻辑复杂的接口、大量数据库读写操作

3）性能测试环境要求

  • 软硬件要求环境一致
    • 线上集群的最小集合
    • 硬件配置最好一致
  • 拒绝多个服务器混部在同一台服务器上：类似 dubbo，一个 provider 最好在一个服务器上，多个 provider 配置在多台服务器上
  • 软件配置和线上保持一致：比如 TomCat 的线程数，JOM 的大小

4）测试数据

  • 用户账号
    • 预估三个月或半年后活跃用户量
    • 避免随便找几十个用户账号就开始测试
  • 参数化数据，数据量、数据大小符合真实分布
  • 设置铺底数据，从镜像库倒入线上数据到测试库
  • 测试脚本和工具：Jmeter、Grinder
  • 性能测试平台：各公司自己搭建

5）测试执行

做好监控：把一切监控起来

  • 性能指标：TPS、MRT、RT90 值、失败率
  • 资源监控：
  • 监控点：CPU、内存、网络、磁盘
  • 监控机器：压测机器、nginx、应用服务、数据库、缓存、消息队列
  • 日志监控：检查错误、异常、及时发现错误
  • Java 监控：堆内存、线程状态、线程数量等

6）流程文档

  • 性能测试准入标准
    • 提测任务描述清楚
    • 功能（第一轮）测试通过
  • 提测需求文档
    • 测试目的、测试内容、测试环境、通过标准、被测系统架构
  • 性能测试通过标准
    • 遗留 bug：无 Major 级别以上 bug
    • 是否符合预期：符合预期指标
