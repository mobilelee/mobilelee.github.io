记录一次k8s下pod启动镜像拉取失败问题排查

发表于2025-08-28|更新于2025-08-28|kubernetes
|浏览量:

现象

镜像在私有harbor仓库，在测试机器k8s上启动pod在创建容器后立马报错ImagePullError，pod一直停留在这个状态

排查过程

1.用describe po命令查看容器获取更多详细信息，内容提示停留在镜像拉取失败，并无更多明显信息

2.尝试用docker pull拉取具体镜像，给出以下报错

3.看上去是认证问题，但由于博主上午刚用该account拉取过harbor仓库的镜像，下午就不行了，用命令get secrets确认了还在，未被删除

4.此时转到了是否是网络问题导致的，通过ping harbor仓库，返回也是联通的状态

5.以上问题都排除了，打算去镜像仓库确认到底这个镜像是否存在，但因为account的账号密码不清楚，需要去xxxxapi-account.yaml文件中寻找，还好使用base64编码的，转一道就得知username和password

6.登陆hargor仓库页面，搜索对应路径下镜像地址，终于破案，镜像已经被删除了，了解到镜像仓库满了，所以把镜像删除，由于是测试环境，并未对镜像进行备份等操作，需要自己去寻找镜像

解决过程

1.首先想到的是去之前部署成功过的node机器上拉取，博主用的是docker image，很遗憾，这个node上也没有该版本镜像了，至此，打算寻求deepseek是否还有其他方式企图能找回镜像，返回的答案推荐尝试以下语句

🔧 方法：使用 ctr 命令（containerd 原生客户端）

ctr 是 containerd 自带的命令行工具，通常默认安装在所有使用 containerd 的节点上。

2.服务器返回了镜像列表,不确定还发给大模型继续帮我确认，他给我的答案也是肯定的，找到了一丝希望

3.将镜像打包并传输至镜像缺失的机器上

4.在机器上讲tar包解压导入并验证是否成功，若tar包较大，需要耐心等待，过一会儿就会down

5.目前这一步镜像已导入，重启pod应该观察结果，但还是报错镜像拉取失败，并无任何更多提示信息，依据过往经验想到了是否可以在yaml里面配置如果本地已有镜像直接拿本地，不去仓库拉取，将配置imagePullPolicy从always改成了IfNotPresent，保存后重启apply- f yaml，终于成功run起来，成功🫰

思考

依据博主过往经验，去找寻镜像的时候均使用docker image去查询，本次不能找到，使用了推荐的ctr -n k8s.io images ls找到了，为何会存在差异，把问题抛向deepsee寻求解答。
简单来说，其实在 Kubernetes 容器运行时，Docker 和 containerd 是两个独立的容器运行时，它们有各自独立的镜像存储和命名空间。由于k8s启动的pod用的是containerd下k8s.io 命名空间，而docker用的是default 命名空间，这个关键区别导致不能用docker image找到但可以在 k8s.io images下找到

扩展阅读

容器运行时的演进

Kubernetes 早期确实直接使用 Docker 作为容器运行时（Container Runtime）。但后来，Kubernetes 推出了 CRI（Container Runtime Interface） 标准，将容器运行时的实现与 Kubernetes 核心解耦。

现在的典型架构是：

Kubernetes → CRI → containerd → 实际容器操作

Docker 本身也是建立在 containerd 之上的：

Docker → containerd

但是关键点在于：Kubernetes 使用的 containerd 和 Docker 使用的 containerd 虽然是同一个软件，但运行在不同的配置和命名空间下。

具体区别

核心概念

containerd 的命名空间
containerd 支持多个独立的命名空间，每个命名空间都有自己完整的镜像和容器集合：

Docker 的命名空间：Docker 使用默认的或特定的命名空间来管理自己的镜像和容器

Kubernetes 的命名空间：Kubernetes 使用 k8s.io 这个特定的命名空间

当你执行：

docker images → 查看的是 Docker 命名空间中的镜像

ctr -n k8s.io images ls → 查看的是 Kubernetes 命名空间中的镜像

好处

隔离性：Kubernetes 管理的容器和 Docker 手动运行的容器互相隔离，不会冲突

稳定性：Kubernetes 不会受到用户手动 Docker 操作的影响

安全性：不同的命名空间提供了额外的安全隔离层

灵活性：可以独立升级或维护某一方的运行时
